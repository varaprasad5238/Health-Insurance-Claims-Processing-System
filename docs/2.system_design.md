# System Design Document

## Plum Health Insurance - AI-Powered Claims Processing System

**Version:** 3.0  
**Date:** June 16, 2026  
**Status:** Implemented Architecture

---

## 1. Executive Summary

This document describes the implemented architecture of the AI-powered OPD claims processing system. The system ingests member-uploaded medical documents, renders PDFs into images, classifies and reads documents with an LLM-backed vision stage, validates required documents through deterministic gating, extracts structured claim data, reconciles amounts, evaluates policy rules from `assignment/policy_terms.json`, and produces an explainable decision with trace reconstruction.

The implementation is a FastAPI backend with a Next.js frontend and an 8-stage operational pipeline: vision reading, document gating, entity extraction, amount reconciliation, orchestration, policy evaluation, decision synthesis, and finalization. Runtime state is stored through SQLAlchemy-backed relational tables and local filesystem artifact storage, with configuration hooks for replacing those local defaults with durable infrastructure.

The design remains multi-agent in structure: each processing stage has a typed contract, emits trace spans, and can be tested independently. LLM calls are centralized through `backend.ai_platform.llm.LLMPlatform`, with support for stub mode, Gemini, OpenAI fallback, retries, circuit-breaker state, JSON parsing, schema validation, and usage metrics.

## 2. Design Philosophy

**Principle 1 - Traceability over convenience.**  
Every major processing stage emits a trace span through `TraceStore.start_span()` and `TraceStore.finish_span()`. A claim's decision history can be reconstructed from the `trace_spans` table and the intermediate JSON outputs written by the trace layer.

**Principle 2 - Policy rules are code, not prompts.**  
The policy engine is deterministic Python code. It loads and validates `assignment/policy_terms.json` through Pydantic models, then evaluates eligibility, waiting periods, covered categories, exclusions, pre-authorization, limits, network discounts, copay, dental/vision item filtering, annual limit, and fraud signals without relying on LLM output for final coverage decisions.

**Principle 3 - Failure is visible and operationally safe.**  
The pipeline catches failures at stage boundaries and routes the claim to `MANUAL_REVIEW` when an extraction, reconciliation, orchestration, or policy stage cannot safely continue. Decision synthesis has a template fallback: if the LLM fails, the system preserves the policy-engine decision and messages instead of losing the result.

**Principle 4 - Use the cheapest correct tool.**  
Vision is used only where image understanding is required. Arithmetic, gating, policy rules, confidence scoring, storage, and trace reconstruction are implemented as Python code. Stub mode is the default for local/demo reliability unless real provider credentials are configured.

**Principle 5 - Deployment pragmatism.**  
The backend keeps infrastructure concerns behind configuration. The database is selected through `DATABASE_URL`, and claim artifact storage is selected through `CLAIMS_STORAGE_ROOT`. This keeps the core architecture independent of any one hosting model.

## 4. LLM Assignment Strategy

All LLM usage goes through `backend.ai_platform.llm.LLMPlatform`. The platform selects a provider based on environment variables, supports stub mode, and records metrics for latency, tokens, fallback usage, status, and errors.

| Stage | Needs vision? | Runtime path | Default/local model behavior | Rationale |
|---|---:|---|---|---|
| Vision reader | Yes | `VisionReaderStage` | Stub by default; Gemini/OpenAI when configured | Classifies documents and returns transcript-like structured output from image bytes. |
| Document gating | No additional call | `DocumentGatingStage` | No LLM | Runs deterministic checks on vision output. |
| Entity extraction | No | `EntityExtractionStage` | Stub by default; configured LLM when enabled | Extracts structured fields from transcripts. |
| Amount reconciler | No | `AmountReconciliationStage` | No LLM | Money parsing and arithmetic checks are deterministic. |
| Orchestrator | No | `ClaimMergeStage` | No LLM | Merges outputs and computes confidence. |
| Policy engine | No | `PolicyEngine` | No LLM | Coverage decisions must be deterministic and auditable. |
| Decision synthesis | No | `DecisionSynthesisStage` | Stub by default; configured LLM when enabled | Produces polished member and ops messages, with template fallback. |

Provider configuration:

```text
USE_STUB_LLM=true          # default local/demo mode
LLM_PROVIDER=stub|gemini|openai
FALLBACK_LLM_PROVIDER=openai
VISION_MODEL=gemini-2.5-flash
FALLBACK_VISION_MODEL=gpt-4o-mini
FAST_MODEL=gemini-2.5-flash
LLM_TIMEOUT_SECONDS=45
LLM_MAX_RETRIES=1
```

Typical claim call count remains cost-conscious: one vision call per uploaded/rendered document page, one entity extraction text call, and one decision synthesis text call. Stub mode makes the demo deterministic and testable without provider credentials.

## 5. Component Inventory and Contracts

### 5.1 Document Gating Agent

**Purpose:** Validate the classified documents before entity extraction and policy evaluation continue.

**Implementation:** `backend/workflow/gating.py`

**Trigger:** After `VisionReaderStage.classify_documents()` returns `DocumentVisionOutput` objects.

**What it does:**
- Loads required documents from `assignment/policy_terms.json`.
- Compares found document types against the required set for the claim category.
- Rejects documents below `READABILITY_THRESHOLD = 0.4`.
- Detects duplicate uploaded document types and reports `WRONG_TYPE` when duplicates cause a required document to be missing.
- Compares patient names across documents with normalized Levenshtein distance.
- Persists gating failure details to `gating_errors`.
- Emits a trace span with `SUCCESS` or `ERROR`.

**Input:**

```python
class GatingRunInput(BaseModel):
    claim_id: str
    claim_category: str
    documents: list[DocumentVisionOutput]
```

**Output (success):**

```python
class GatingPassed(BaseModel):
    passed: Literal[True] = True
    docs_validated: int
    patient_name_match: bool
    required_docs: list[str]
    found_docs: list[str]
    patient_names: list[str] = []
```

**Output (failure):**

```python
class GatingFailed(BaseModel):
    passed: Literal[False] = False
    error_code: Literal["WRONG_TYPE", "UNREADABLE", "PATIENT_MISMATCH", "MISSING_REQUIRED"]
    human_message: str
    detail: dict[str, Any]
```

### 5.2 Vision Reading Agent

**Purpose:** Classify each uploaded document image and extract transcript/readability metadata.

**Implementation:** `backend/workflow/vision_reader.py`

**Architecture:** The backend preprocessor returns image bytes for uploaded images or rendered PDF pages. `VisionReaderStage` classifies documents in parallel with `asyncio.gather`, one LLM platform call per document payload. The parsed output is a `DocumentVisionListOutput`, which can contain one or more logical documents.

**Input:**

```python
class VisionDocumentPayload(TypedDict):
    file_name: str
    content_type: str
    raw_bytes: bytes
    size_bytes: int
    stored_path: str
    source_page_range: str
    source_upload_index: int
```

**Output:**

```python
class DocumentVisionOutput(BaseModel):
    document_type: Literal[
        "PRESCRIPTION", "HOSPITAL_BILL", "LAB_REPORT", "PHARMACY_BILL",
        "DENTAL_REPORT", "DISCHARGE_SUMMARY", "UNKNOWN"
    ]
    confidence: float
    readability: float
    patient_name_raw: str | None = None
    quality_flags: list[QualityFlag] = []
    transcript: str
    source_file_name: str | None = None
    source_page_range: str | None = None
```

**Errors:** Validation failures and provider failures are traced as `ERROR`. In the ingestion pipeline, a vision classification exception routes the claim to `MANUAL_REVIEW`.

### 5.4 Amount Reconciler Agent

**Purpose:** Reconcile claimed amount, extracted bill total, and line item sum.

**Implementation:** `backend/workflow/reconciler.py`

**What it checks:**
- Whether extracted bill total differs from the line item sum by more than 1.00.
- Whether claimed amount differs from the best document amount by more than 1.00.
- Whether no reliable amount was extracted.
- Whether extraction uncertainty text suggests alteration/correction.

**Output:**

```python
class AmountReconciliationResult(BaseModel):
    bill_total_extracted: str | None = None
    line_items_sum: str | None = None
    claimed_amount: str
    payable_basis_amount: str
    discrepancy_flags: list[DiscrepancyFlag] = []
    fraud_indicators: list[FraudIndicator] = []
    agent_status: str = "SUCCESS"
```

The payable basis is the lower of the claimed amount and document amount when both are available; otherwise it falls back to claimed amount.

### 5.5 Orchestrator

**Purpose:** Merge classified documents, extraction output, and reconciliation output into a single `MergedClaimResult` for policy evaluation.

**Implementation:** `backend/workflow/orchestrator.py`

**Confidence formula:**

```text
document_confidence = average(confidence * 0.60 + readability * 0.40)
entity_confidence = weighted_average(field_confidences)
reconciliation_confidence = max(0.0, 1.0 - 0.12 * discrepancies - 0.18 * fraud_indicators)

extraction_confidence =
    document_confidence * 0.35
  + entity_confidence * 0.45
  + reconciliation_confidence * 0.20
  - 0.15 * failed_agents
  - 0.05 * discrepancy_count
  - 0.08 * fraud_indicator_count
```

**Output:**

```python
class MergedClaimResult(BaseModel):
    patient_name: str | None = None
    doctor_name: str | None = None
    doctor_registration: str | None = None
    diagnosis_primary: str | None = None
    treatment_date: str | None = None
    hospital_name: str | None = None
    line_items: list[LineItemOutput] = []
    extracted_total_amount: str | None = None
    claimed_amount: str
    payable_basis_amount: str
    extraction_confidence: float
    failed_agents: list[str] = []
    failed_stages: list[str] = []
    conflict_log: list[ConflictEntry] = []
    discrepancy_flags: list[dict] = []
    fraud_indicators: list[dict] = []
    document_confidence: float | None = None
    entity_confidence: float | None = None
    reconciliation_confidence: float | None = None
```

Claims with `extraction_confidence < 0.65` are routed to `MANUAL_REVIEW` before policy evaluation.

### 5.6 Policy Rule Engine (Deterministic - No LLM)

**Purpose:** Decide payable amount and final coverage outcome using policy terms and merged claim data.

**Implementation:** `backend/policy/engine.py`

The rule flow is implemented centrally in `PolicyEngine.evaluate_sync()`. The `backend/policy/rules/` modules exist as logical rule-group placeholders, but the active runtime path is the central engine.

```python
class RuleResult(BaseModel):
    rule_id: str
    outcome: Literal["PASS", "FAIL", "PARTIAL", "SKIP"]
    reason: str
    approved_amount: str | None = None
    deducted_amount: str | None = None
    deduction_reason: str | None = None
    metadata: dict = {}
```

**Implemented ordered rule flow:**

| # | Rule | Runtime rule ID | Behavior |
|---:|---|---|---|
| 1 | Member eligibility | `MEMBER_ELIGIBILITY` | Reject unknown member. |
| 2 | Policy active date | `POLICY_ACTIVE` | Reject treatment outside policy period. |
| 3 | Minimum claim amount | `MINIMUM_CLAIM_AMOUNT` | Reject below policy minimum. |
| 4 | Initial waiting period | `INITIAL_WAITING_PERIOD` | Reject before join date + initial wait. |
| 5 | Condition waiting period | `CONDITION_WAITING_PERIOD` | Reject matching diagnosis before condition-specific eligibility date. |
| 6 | Category coverage | `COVERAGE_CATEGORY` | Reject uncovered/unknown claim category. |
| 7 | Dental line item filter | `DENTAL_LINE_ITEM_FILTER` | Approve/reject dental items and support `PARTIAL`. |
| 8 | Vision line item filter | `VISION_LINE_ITEM_FILTER` | Approve/reject vision items and support `PARTIAL`. |
| 9 | Covered amount check | `COVERED_AMOUNT` | Reject if no covered amount remains. |
| 10 | Exclusion check | `EXCLUSION_CHECK` | Reject excluded treatment when no item-level partial filter applies. |
| 11 | Pre-authorization | `PRE_AUTH_CHECK` | Reject high-value diagnostic tests without pre-auth signal. |
| 12 | Same-day fraud signal | `FRAUD_SIGNAL_CHECK` | Route to `MANUAL_REVIEW` if same-day count exceeds threshold. |
| 13 | Benefit cap | `BENEFIT_CAP` | Reject when payable amount exceeds category sub-limit or per-claim cap. |
| 14 | Annual OPD limit | `ANNUAL_LIMIT` | Reject if YTD + current exceeds annual limit; skip if YTD absent. |
| 15 | Network discount | `NETWORK_DISCOUNT` | Apply category network discount when hospital matches. |
| 16 | Copay | `COPAY_APPLICATION` | Apply category copay after network discount and return final decision. |

**Not currently implemented:** submission-deadline enforcement and monthly claim fraud threshold. These remain future rules.

### 5.7 Decision Synthesis Agent

**Purpose:** Produce member-facing and operations-facing text from a policy decision and merged claim.

**Implementation:** `backend/workflow/decision.py`

This stage does not decide coverage. It calls the configured LLM provider with `DECISION_SYNTHESIS_PROMPT` and parses a `DecisionMessageOutput`:

```python
class DecisionMessageOutput(BaseModel):
    member_message: str
    ops_summary: str
```

If the LLM call or schema parsing fails, the stage returns the original `PolicyDecisionResult` generated by the policy engine and records a `PARTIAL` trace span with `template_fallback_used=True`.

### 5.8 Trace Emitter (Cross-Cutting)

**Implementation:** `backend/tracing/store.py` and `backend/tracing/span.py`

The implementation uses explicit trace-store calls rather than a decorator. Each traced stage records:

```python
class TraceSpanDTO(BaseModel):
    span_id: str
    agent_name: str
    stage_name: str | None
    stage_order: int
    status: str
    elapsed_ms: int | None
    started_at: str
    ended_at: str | None
    input_summary: Any | None
    output_summary: Any | None
    confidence_delta: float | None
    errors: list[Any]
    model_used: str | None
```

Trace data is persisted in SQLite through `TraceSpanModel`. `finish_span()` also writes an intermediate JSON artifact through `backend.storage.local.write_intermediate_output()`.

Trace JSON artifacts are written under the configured claim artifact root. In a durable deployment, this storage root should point to persistent shared storage or object storage rather than process-local disk.

## 8. Confidence Model

Confidence is computed by the orchestrator. It combines document quality, entity confidence, reconciliation confidence, and penalties.

```text
DOCUMENT CONFIDENCE
    average(document.confidence * 0.60 + document.readability * 0.40)

ENTITY CONFIDENCE
    weighted field confidence:
      total_amount or amount = 0.30
      patient_name = 0.25
      diagnosis_primary or diagnosis = 0.20
      doctor_name or doctor_registration = 0.15
      treatment_date = 0.10

RECONCILIATION CONFIDENCE
    1.0 - 0.12 * discrepancy_count - 0.18 * fraud_indicator_count

EXTRACTION CONFIDENCE
    document_confidence * 0.35
  + entity_confidence * 0.45
  + reconciliation_confidence * 0.20
  - 0.15 * failed_agent_count
  - 0.05 * discrepancy_count
  - 0.08 * fraud_indicator_count

THRESHOLD
    < 0.65  => MANUAL_REVIEW
    >= 0.65 => policy evaluation continues
```

The 0.65 threshold is currently hardcoded in `run_claim_ingestion()`. A separate caution flag for 0.65-0.85 is not currently persisted.

## 9. Failure Handling Matrix

| Failure type | Current system behavior | Trace record | Claim status impact |
|---|---|---|---|
| No documents or too many documents | `VisionReaderStage` raises `ValueError` | Vision error when reached through ingestion | `MANUAL_REVIEW` |
| Vision LLM/schema/provider error | Exception caught by ingestion | Vision span `ERROR` when span started | `MANUAL_REVIEW` |
| Gating failure | Persist `GatingErrorModel`; finish gating span with `ERROR` | `gating` span `ERROR` | `GATING_FAILED` |
| Entity extraction error | Exception caught by ingestion | Entity span `ERROR` | `MANUAL_REVIEW` |
| Amount reconciliation error | Exception caught by ingestion | Reconciler span `ERROR` | `MANUAL_REVIEW` |
| Orchestration error | Exception caught by ingestion | Orchestrator span `ERROR` | `MANUAL_REVIEW` |
| Low extraction confidence | Pipeline stops before policy engine | Merge span shows low confidence | `MANUAL_REVIEW` |
| Policy engine exception | Exception caught by ingestion | Policy span `ERROR` if started | `MANUAL_REVIEW` |
| Decision synthesis LLM failure | Keep policy-engine decision and template text | `decision_synthesis` span `PARTIAL` | Decision still persisted |
| Trace JSON artifact write failure | Swallowed inside `finish_span()` | DB span still persists | No claim-status impact |

## 13. Scale Considerations (75K -> 10M Lives)

The current implementation is optimized for an assignment/demo workload and a simple single-service backend. It is not yet a durable high-scale production architecture.

**Current demo/runtime characteristics:**
- BackgroundTasks runs work in the API process instead of a distributed queue.
- SQLite or a configured relational database stores claims, traces, decisions, members, policy seed data, gating errors, and LLM metrics.
- Claim upload artifacts and intermediate trace JSON files use the configured filesystem storage root.
- LLM calls are centralized and can retry/fallback, but provider-level throughput must still be configured externally.

**Bottlenecks at scale:**
- Vision LLM calls dominate latency and cost.
- Process-local SQLite and process-local artifact storage are unsuitable for durable multi-instance storage.
- BackgroundTasks does not provide retry queues, dead-letter queues, or worker isolation.
- Local artifact storage cannot survive execution environment replacement.
- Trace rows and LLM metrics need indexing, partitioning, and retention policies.

**Production evolution path:**
- Replace local SQLite with PostgreSQL or another managed relational database using `DATABASE_URL`.
- Replace local artifact storage with object storage or shared filesystem storage using `CLAIMS_STORAGE_ROOT` as the migration point.
- Replace BackgroundTasks with a dedicated queue or workflow engine for explicit stage orchestration.
- Store trace artifacts in durable storage and keep DB rows as the queryable index.
- Add request authentication, member authorization, and tenant boundaries.
- Add monthly fraud rolling windows and durable claim-history aggregation.
- Add provider rate-limit controls, prompt caching, and model routing by claim category.
- Add operational dashboards and alarms for failures, latency, fallback rate, and token usage.

## 14. Conscious Trade-offs

**Included in implementation:**
- FastAPI backend with typed DTOs and SQLAlchemy persistence.
- Multi-stage claim-processing pipeline with trace spans.
- Deterministic policy engine from `policy_terms.json`.
- LLM platform with stub, Gemini, OpenAI fallback, retries, circuit state, schema parsing, and metrics.
- API endpoints for policies/members, claim submission, claim listing, claim status, test-suite execution, and LLM metrics.
- Next.js frontend routes for submit, claims, claim detail, test suite, and LLM metrics.
- pytest coverage configuration with an enforced 80% backend threshold.

**Excluded or deferred:**
- Durable production database provisioning.
- Durable object/shared storage for claim artifacts.
- Celery/Redis, a message queue, a workflow engine, or separate worker process.
- Multi-tenant authentication and authorization.
- Full rule-module decomposition under `backend/policy/rules`.
- Submission deadline rule.
- Monthly claim fraud threshold rule.
- Explicit 0.65-0.85 caution flag persistence.
- Claude provider implementation.
- Dedicated `@traced` decorator; tracing is explicit through `TraceStore` calls.

**Verified state:**
- Backend tests: 57 passed.
- Coverage: above 81% with an 80% gate.
